# ADR-002: Infraestructura de datos con S3

## Identificación

**Número**: ADR-002  
**Fecha**: 2026-03-20  
**Redactor**: @creep1ng, Ricardo Arias  
**Estado**: Aceptado

---

## Contexto

El proyecto requiere un sistema estandarizado y confiable para almacenar, acceder y sincronizar las fichas técnicas de productos (paneles solares, inversores, controladores, baterías) utilizadas por el chatbot RAG. El almacenamiento local en Nextcloud presenta inestabilidad recurrente, lo cual afecta negativamente la reproducibilidad del entorno de desarrollo y los pipelines de CI. Se requiere una solución que funcione de manera consistente en múltiples contextos: desarrollo local, CI en GitHub Actions, y potencialmente despliegues en cloud.

El volumen de datos inicial es moderado (≤50 fichas técnicas según US-09), pero una arquitectura que permita escalar es requerida. Adicionalmente, el sistema debe mantener un mirror de los documentos que permita tomar decisiones técnicas específicas para el chatbot sin afectar la lógica de negocio de los documentos en producción en Nextcloud.

---

## Decisión

Se adopta **Amazon S3** como almacenamiento primario para las fichas técnicas del chatbot, siendo utilizada la siguiente estructura de buckets:

```
arte-chatbot-data/
├── raw/
│   ├── paneles/
│   ├── inversores/
│   ├── controladores/
│   └── baterias/
└── index/
    └── catalog_index.json
```

El acceso a los archivos es realizado mediante el SDK de AWS (boto3), utilizando credenciales configuradas mediante variables de entorno:

- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `AWS_BUCKET_NAME`

Esta infraestructura sirve como base para el sistema de File Inputs del chatbot (documentado en ADR-003) y el pipeline de indexación mencionado en TS-03.

## Justificación

### Alternativas consideradas

1. **Nextcloud con acceso WebDAV**: Descartado por cuestiones de seguridad (credenciales compartidas, falta de control de acceso granular) e incertidumbre en la disponibilidad del servicio. Fue experimentado por el equipo inestabilidad recurrente que afectaba el desarrollo.

2. **S3 compatible de Hetzner (Object Storage)**: Descartado por tres razones principales: (a) ausencia de free tier, (b) servidores físicamente ubicados en Alemania/Finlandia, lejos de la región Latinoamérica donde opera el negocio, lo cual incrementa latencia, y (c) preferencia explícita de los stakeholders a favor de AWS.

3. **Almacenamiento local en cada máquina de desarrollo**: Descartado debido a que genera problemas de normalización y sincronización entre los miembros del equipo. Cada desarrollador持有一ría una versión diferente de los archivos, complicando la reproducibilidad y haciendo inviable el entorno de CI.

### Criterios de decisión

- **Estabilidad y disponibilidad**: AWS S3 ofrece >99.99% de disponibilidad, eliminando los problemas recurrentes experimentados con Nextcloud.
- **Consistencia entre entornos**: Un bucket compartido garantiza que todos los desarrolladores, CI y ambientes cloud trabajen con los mismos datos.
- **Latencia**: La región us-west de AWS está geográficamente más cercana a Latinoamérica que las alternativas europeas evaluadas.
- **Seguridad**: Mediante IAM policies pueden definirse credenciales de mínimo privilegio específicas para el proyecto.
- **Escalabilidad**: S3 escala automáticamente sin intervención, soportando desde el prototype actual hasta producción.

### Trade-offs aceptados

- **Lock-in con AWS**: Una dependencia con Amazon S3 existe en las fases iniciales. Es mitigable mediante abstracciones en el código (protocolo de acceso) que permitan migrar a otro proveedor S3-compatible si fuera necesario.
- **Riesgo de caída de AWS**: La disponibilidad de S3 es alta pero no absoluta. Para el alcance académico actual, el riesgo es aceptable.
- **Overhead de configuración**: Cada desarrollador debe configurar credenciales AWS y el equipo de infra debe gestionar su rotación y seguridad.

## Consecuencias

### Positivas

- **Forma estandarizada de acceso a datos**: Todos los entornos (local, CI, cloud) utilizan el mismo mecanismo y los mismos datos.
- **Alta disponibilidad**: AWS S3 garantiza acceso constante a las fichas técnicas sin las caídas recurrentes experimentadas con Nextcloud.
- **Escalabilidad transparente**: El almacenamiento escala automáticamente sin cambios en el código.
- **Control de seguridad granular**: Las IAM policies permiten limitar el acceso a solo lectura del bucket específico, reduciendo la superficie de ataque.
- **Compatibilidad con múltiples herramientas**: S3 es ampliamente soportado por libraries, CLIs y servicios de CI/CD.

### Negativas

- **Lock-in con Amazon Web Services**: El proyecto queda dependent de AWS S3. Una migración futura a otro proveedor requeriría cambios en el código de ingesta y acceso.
- **Riesgo de caída de AWS**: Aunque improbable, una outage de S3 afectaría directamente al chatbot. Para producción futura debería considerarse replication cross-region.
- **Overhead técnico adicional**: La implementación requiere boto3 como dependencia, configuración de credenciales AWS por cada desarrollador, y mantención de la seguridad de dichas credenciales por parte del equipo de infraestructura.

### Riesgos identificados

- **Riesgo**: Exposición de credenciales AWS en el repositorio.
  **Mitigación**: Las credenciales son gestionadas exclusivamente via variables de entorno y `.env` (ya en `.gitignore`). Rotación periódica de keys mediante IAM es realizada.
- **Riesgo**: Costo inesperado de S3 por crecimiento no anticipado.
  **Mitigación**: Con ≤50 fichas técnicas (PDFs de <10MB cada uno), el costo mensual será marginal (<1 USD/mes). Monitoreo es realizado con AWS Budgets.
- **Riesgo**: Latencia elevada en desarrollo local por distancia geográfica al bucket.
  **Mitigación**: Para desarrollo local, considerar caching local de fichas técnicas más consultadas.

## Notas adicionales

### Estructura de carpetas future-proof

La estructura `raw/` + `index/` permite extender el sistema sin modificar el acceso existente. El directorio `index/` almacenará el archivo `catalog_index.json` mencionado en TS-03, el cual será generado automáticamente por el pipeline de ingesta y utilizado para el System Prompt del LLM.

### Abstracción del proveedor de almacenamiento

El código de acceso a datos debe ser implementado siguiendo el patrón de abstracción (ver SOLID - Open/Closed Principle), de modo que el proveedor de almacenamiento pueda intercambiarse sin modificar la lógica de negocio del RAG. Una interfaz `DataProvider` será definida con métodos `list_documents()`, `get_document()`, etc.

### Consideraciones para producción futura

Si el proyecto escala a producción, deberá evaluarse:
- S3 Replication a otra región para disaster recovery
- CloudFront como CDN para reducir latencia
- S3 Intelligent-Tiering para optimizar costos de almacenamiento a largo plazo
