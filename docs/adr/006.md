# ADR-006: Búsqueda de productos y resolución de ruta S3 en Tool Calling

## Identificación

**Número**: ADR-006
**Fecha**: 2026-03-26
**Redactor**: Arte Chatbot Team
**Estado**: Propuesto

---

## Contexto

### Situación actual

El chatbot de Arte Soluciones Energéticas implementa Tool Calling mediante la herramienta `leer_ficha_tecnica`, que permite al LLM consultar fichas técnicas de productos almacenados como PDFs en S3. Sin embargo, existe una contradicción entre las capas del sistema que causa un error en producción:

| Capa | Comportamiento |
|---|---|
| **Tool schema** (`backend/app/tools.py`) | `ruta_s3` es **opcional**; solo `categoria` es requerida. `strict: False` |
| **System prompt** (`backend/app/llm_client.py`) | Instruye al LLM: *"Puedes invocarla con datos parciales... Si faltan campos como fabricante o modelo, déjalos vacíos. El sistema buscará en el catálogo."* |
| **Backend** (`backend/main.py:129-130`) | `if not ruta_s3: raise ValueError("Missing ruta_s3 in tool arguments")` — **exige** `ruta_s3` |

Esto genera el error `ValueError: Missing ruta_s3 in tool arguments` cuando el LLM, siguiendo las instrucciones del prompt, invoca la herramienta sin conocer la ruta S3 exacta.

Además, el sistema no soporta dos casos de uso fundamentales:

1. **Consulta técnica específica** (ej: *"¿Cuántos paneles puedo conectarle al inversor Greenpoint de 3000W?"*): El usuario identifica un producto pero el LLM no puede resolver la ruta S3 correspondiente.

2. **Búsqueda de alternativas** (ej: *"¿Qué inversores puedo usar para un sistema solar de 2000W?"*): El usuario no identifica un producto específico. No existe mecanismo para que el LLM descubra productos que cumplan criterios y ofrezca alternativas.

### Fuerzas en juego

- **Experiencia del usuario**: Los clientes B2B esperan respuestas que incluyan tanto recomendaciones de productos como datos técnicos verificables de fichas reales.
- **Catálogo estructurado**: El proyecto ya genera y mantiene un `catalog_index.json` con metadatos de productos (fabricante, modelo, categoría, capacidad, ruta S3), pero este índice no es accesible para el LLM durante la conversación.
- **Patrón de Tool Calling estándar**: OpenAI y la industria en general recomiendan el patrón de múltiples herramientas (buscar → leer) con loops agenticos para este tipo de flujos.
- **Escalabilidad**: A medida que crece el catálogo de productos, el sistema debe poder buscar dinámicamente sin requerir actualizaciones manuales del prompt.
- **Compatibilidad hacia atrás**: Las llamadas existentes que proveen `ruta_s3` directamente deben seguir funcionando.

### Alcance

Esta decisión afecta a:
- `backend/app/tools.py` — Nueva tool `buscar_producto`
- `backend/app/catalog.py` — Nuevo módulo de búsqueda sobre el índice
- `backend/main.py` — Loop agentic y handler de búsqueda
- `backend/app/llm_client.py` — Actualización del system prompt
- `backend/tests/` — Tests nuevos y actualizados

---

## Decisión

Vamos a implementar una segunda herramienta `buscar_producto` que permite al LLM buscar productos en el catálogo por criterios (categoría, fabricante, capacidad, tipo), combinada con un loop agentic que soporta múltiples iteraciones de tool calling, porque esto resuelve ambos casos de uso de forma natural, separa las responsabilidades de búsqueda y lectura, y sigue el patrón estándar de agentes (Plan → Search → Read → Respond).

---

## Justificación

### Alternativas consideradas

#### Alternativa A: Inyectar el catálogo en el system prompt
Cargar el contenido de `catalog_index.json` directamente en el `ARTE_SYSTEM_PROMPT` como contexto estático.

- **Pro**: Mínimo cambio arquitectónico, el LLM tiene contexto inmediato.
- **Contra**: Prompt crece significativamente por cada producto agregado. No escalable. El LLM puede alucinar rutas S3 incorrectas. Requiere actualización manual del prompt al cambiar el catálogo.
- **Decisión**: Descartada por problemas de escalabilidad y mantenimiento.

#### Alternativa C: Modificar `leer_ficha_tecnica` para manejar búsqueda internamente
Una sola tool con dos comportamientos: si hay `ruta_s3` lee el PDF; si no, busca en el catálogo.

- **Pro**: Una sola tool (menos tokens). Backward compatible.
- **Contra**: Una tool con dos comportamientos distintos confunde al LLM. El resultado varía (PDF vs lista de productos). Más difícil de testear.
- **Decisión**: Descartada por violar el principio de responsabilidad única.

#### Alternativa E (B+D): Combinación de buscar_producto + loop agentic (SELECCIONADA)
Agregar tool separada de búsqueda y loop agentic que permita al LLM encadenar múltiples tool calls.

- **Pro**: Separación clara de responsabilidades. Patrón estándar probado. Máxima flexibilidad. Escalable a nuevas herramientas.
- **Contra**: Mayor cambio arquitectónico. Más llamadas LLM (costo/latencia). Complejidad de debugging.
- **Decisión**: Seleccionada por ser la más robusta y escalable.

### Criterios de decisión

1. **Escalabilidad**: La solución debe soportar un catálogo creciente sin cambios manuales en el prompt.
2. **Separación de responsabilidades**: Buscar productos y leer fichas técnicas son operaciones distintas.
3. **Compatibilidad**: Las llamadas existentes con `ruta_s3` directo deben seguir funcionando.
4. **Estándares de la industria**: El patrón de múltiples tools con loop agentic es el estándar recomendado por OpenAI.
5. **Complejidad aceptable**: La complejidad adicional se justifica por la flexibilidad ganada.

### Trade-offs aceptados

- **Latencia aumentada**: CU1 pasa de 2 a 3 llamadas LLM; CU2 puede requerir 3-6 llamadas. Aceptable para un chatbot B2B donde la precisión es prioritaria sobre la velocidad.
- **Costo por consulta**: ~1.5-3x el costo actual por consulta con búsqueda. Aceptable dado el volumen moderado esperado.
- **Complejidad del backend**: El loop agentic agrega complejidad al handler `/chat`. Mitigado con un límite de iteraciones y logging detallado.

## Consecuencias

### Positivas

- **Resolución de ambos casos de uso**: El sistema puede tanto responder consultas técnicas específicas como ofrecer alternativas de productos.
- **Eliminación del error `Missing ruta_s3`**: El backend ya no lanza excepciones cuando el LLM no conoce la ruta S3.
- **Descubrimiento de productos**: El LLM puede descubrir productos en el catálogo sin hardcodear rutas.
- **Escalabilidad**: Agregar productos al catálogo no requiere cambios en el prompt ni en el código.
- **Separación clara de concerns**: `buscar_producto` para discovery, `leer_ficha_tecnica` para lectura de documentos.
- **Extensibilidad**: El loop agentic permite agregar más herramientas en el futuro (ej: calcular presupuesto, consultar inventario).

### Negativas

- **Latencia adicional**: Las consultas con búsqueda tendrán 1-3 llamadas LLM adicionales.
- **Costo por consulta**: Aproximadamente 1.5-3x el costo actual cuando se usa búsqueda.
- **Complejidad de debugging**: El loop agentic hace que el flujo sea menos lineal y más difícil de depurar.
- **Nuevo módulo**: `catalog.py` agrega código que debe mantenerse sincronizado con el formato de `catalog_index.json`.

### Riesgos identificados

- **Riesgo**: El LLM entre en un loop infinito de tool calls.
  **Mitigación**: Límite máximo de iteraciones configurable (default: 5) con logging de cada iteración.

- **Riesgo**: La búsqueda retorne demasiados productos y el LLM no pueda procesarlos.
  **Mitigación**: Limitar resultados a un máximo configurable (default: 10) y ordenar por relevancia.

- **Riesgo**: El catálogo `catalog_index.json` se desincroniza con los archivos reales en S3.
  **Mitigación**: Este riesgo ya existe en la arquitectura actual. El pipeline de generación de índice ya maneja la sincronización.

- **Riesgo**: El modelo `gpt-5.4-nano` no maneja bien múltiples tools.
  **Mitigación**: Verificar compatibilidad durante la implementación. Si no funciona, usar `gpt-4o-mini` como fallback.

## Notas adicionales

### Sobre la compatibilidad del modelo

El modelo actual `gpt-5.4-nano` debe soportar múltiples tools simultáneas. La experiencia de OpenAI con modelos nano sugiere que el manejo de 2 tools es confiable. Si se detectan problemas de selección de tools durante la implementación, se evaluará cambiar a `gpt-4o-mini`.

### Sobre el módulo `catalog.py`

Se recomienda que este módulo implemente una interfaz abstracta (`BaseCatalogSearch`) para facilitar la sustitución del backend de búsqueda en el futuro (ej: migrar de búsqueda en JSON a base de datos si el catálogo crece significativamente).

### Sobre la iteración incremental

La implementación se recomienda en dos fases:
1. **Fase 1**: Agregar `buscar_producto` con búsqueda básica + corregir el `ValueError` de `ruta_s3`.
2. **Fase 2**: Implementar el loop agentic para soportar múltiples tool calls en secuencia.

Esto permite validar la tool de búsqueda independientemente antes de introducir la complejidad del loop agentic.
