# ADR-007: Orquestador Central de Evaluación

## Status

Implemented

## Context

El proyecto cuenta actualmente con múltiples *evaluation harnesses* independientes:
- `evaluation/harness/run.py`
- `evaluation/intent_eval/run_eval.py`
- `evaluation/escalation_eval/run_eval.py`
- `evaluation/hallucination_check.py`
- `evaluation/human_eval/cli.py`

Cada uno:
- Tiene su propio dataset JSON con esquemas distintos
- Invoca la API `/chat` independientemente (3× costo y latencia)
- Duplica código: HTTP client, health check, semáforo, formato de salida
- No comparte resultados entre sí
- `human_eval` usa datos estáticos en lugar de respuestas generadas en tiempo real

## Problems

1. **Código duplicado**: HTTP client, health check, semáforo, CSV/JSON save repetidos
2. **Múltiples datasets**: Cada eval tiene su schema, imposibilitando correlación
3. **Invocaciones redundantes**: Cada eval llama a `/chat` por separado → costo 3×, latencia 3×
4. **Sin early return**: Si el harness principal falla, los demás siguen ejecutándose en CI
5. **Human eval desacoplado**: Opera sobre conversaciones pre-grabadas, no reutiliza las respuestas reales del chatbot

## Decision

Implementar un **Orquestador Central de Evaluación** que:

1. Carga un **dataset unificado** (`evaluation/dataset.json`) con todos los campos necesarios
2. Ejecuta **una sola pasada** de queries contra el backend (`POST /chat`)
3. Guarda `raw_results.json` con todas las respuestas completas
4. Ejecuta **Analyzers** en serie sobre los mismos resultados:
   - `GeneralAnalyzer` → latencia, éxito/fallo
   - `IntentAnalyzer` → accuracy de `intent_type` (US-05)
   - `EscalationAnalyzer` → FP/FN rates (US-10)
   - `HallucinationAnalyzer` → tasa de alucinación (US-07)
5. Genera **reporte consolidado** (JSON)
6. Sube artefactos a **S3** bajo `evaluation/orchestrator/{commit}_{ts}.json`
7. Implementa **early return** ante fallos críticos (health check fallido, todas las queries fallan)

## Implementation

```
evaluation/
├── orchestrator/
│   ├── __init__.py
│   ├── run.py              # Entrypoint principal
│   ├── executor.py         # Ejecutor de queries con httpx.AsyncClient
│   ├── settings.py         # Configuración (Pydantic)
│   ├── upload_to_s3.sh     # Script bash para subida a S3
│   └── analyzers/
│       ├── __init__.py
│       ├── base.py         # BaseAnalyzer (interfaz)
│       ├── general.py      # Métricas generales
│       ├── intent.py       # US-05: intent classification
│       ├── escalation.py   # US-10: escalation detection
│       └── hallucination.py # US-07: hallucination detection
├── dataset.json            # Dataset unificado (merge de todos)
├── s3_client.py            # Cliente S3 existente
└── storage.py             # Utilidades de almacenamiento existentes
```

## Usage

```bash
# Ejecutar orchestrator completo
python -m evaluation.orchestrator.run

# Ejecutar sin subir a S3
python -m evaluation.orchestrator.run --no-upload

# Usar resultados crudos de una ejecución anterior
python -m evaluation.orchestrator.run --from-raw path/to/raw_results.json

# Subir resultado a S3 manualmente
./evaluation/orchestrator/upload_to_s3.sh
```

## Benefits

- **Reducción de costo**: 1 pasada vs. 3–5 pasadas independientes
- **Consistencia**: Mismas respuestas para todos los análisis
- **Early return**: Si el backend falla, la evaluación termina inmediatamente
- **Extensibilidad**: Nuevos analyzers se agregan sin modificar el orquestador
- **Human eval integrado**: Puede calificar respuestas reales del chatbot
- **CI/CD simplificado**: Un solo comando, un solo reporte

## Trade-offs

- Mayor complejidad inicial en el orquestador
- Acoplamiento temporal entre ejecución y análisis (roto por `--from-raw`)
- Requiere migración de datasets existentes
