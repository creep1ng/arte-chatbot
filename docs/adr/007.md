# ADR-007: Migración a Async I/O con AsyncOpenAI

## Identificación

**Número**: ADR-007
**Fecha**: 2026-03-29
**Redactor**: Arte Chatbot Team
**Estado**: Propuesto

---

## Contexto

### Situación actual

El backend del chatbot utiliza clientes sincrónicos para todas las operaciones de I/O:

| Componente | Cliente actual | Comportamiento |
|---|---|---|
| LLM (OpenAI Chat Completions) | `openai.OpenAI` | Bloquea el thread durante 2-8s por llamada |
| File Inputs (OpenAI Files API) | `openai.OpenAI` | Bloquea durante upload/delete (~1-3s) |
| Almacenamiento (AWS S3) | `boto3.client` | Bloquea durante descarga (~0.5-2s) |

El endpoint `/chat` es una función sincrónica (`def`, no `async def`), por lo que FastAPI lo ejecuta en un threadpool de ~40 threads por defecto. Cada request mantiene un thread bloqueado durante todo su procesamiento (5-20 segundos), lo que limita severamente la capacidad de atender múltiples clientes concurrentes.

### Fuerzas en juego

- **Concurrencia limitada**: Con requests de ~10s promedio y 40 threads, el throughput máximo teórico es ~4 req/s. En la práctica, por contention de GIL y overhead de threading, es menor.
- **I/O bound**: Las operaciones que dominan la latencia son llamadas de red (OpenAI, S3), no cómputo. Los threads están ociosos la mayor parte del tiempo esperando respuestas.
- **OpenAI SDK soporta async nativamente**: El SDK `openai>=1.68.0` incluye `AsyncOpenAI` con la misma API que `OpenAI`, haciendo la migración transparente.
- **FastAPI es async-first**: FastAPI tiene soporte nativo para `async def` endpoints, con mejor rendimiento que endpoints sync delegados a threadpool.

### Alcance

Esta decisión afecta a:
- `backend/app/llm_client.py` — Migración de `OpenAI` a `AsyncOpenAI`
- `backend/app/file_inputs.py` — Migración de `OpenAI` a `AsyncOpenAI`
- `backend/app/s3_client.py` — Wrapper async sobre `boto3` sync
- `backend/app/session.py` — `threading.Lock` → `asyncio.Lock`
- Todos los archivos de test que mockean estos clientes

---

## Decisión

Vamos a migrar todos los clientes de I/O de sincrónicos a asíncronos, utilizando `AsyncOpenAI` del SDK `openai` para las llamadas a OpenAI y `asyncio.to_thread()` como wrapper para las operaciones S3 con `boto3`, porque esto permite que el event loop de asyncio gestione múltiples requests concurrentes sin bloquear, aprovechando la naturaleza I/O-bound del sistema.

### Estrategia por componente

#### LLMClient: `OpenAI` → `AsyncOpenAI`

```python
# Antes
from openai import OpenAI
self._openai_client: Optional[OpenAI] = None
response = self.openai_client.chat.completions.create(...)

# Después
from openai import AsyncOpenAI
self._openai_client: Optional[AsyncOpenAI] = None
response = await self.openai_client.chat.completions.create(...)
```

La API del SDK es idéntica entre `OpenAI` y `AsyncOpenAI`. La única diferencia es que los métodos retornan coroutines que deben ser awaited.

#### FileInputsClient: `OpenAI` → `AsyncOpenAI`

Mismo patrón que LLMClient. `upload_pdf()` y `delete_file()` se convierten en `async def`.

#### S3Client: `asyncio.to_thread()` wrapper

```python
import asyncio

async def download_pdf(self, s3_key: str) -> bytes:
    """Async wrapper que ejecuta la descarga sync en un thread."""
    return await asyncio.to_thread(self._download_pdf_sync, s3_key)

def _download_pdf_sync(self, s3_key: str) -> bytes:
    # Implementación actual de boto3 sin cambios
    ...
```

**No se agrega `aioboto3`** como dependencia porque:
1. Las operaciones S3 son rápidas (~100-500ms), no justifican la complejidad adicional
2. `asyncio.to_thread()` es parte de la stdlib de Python 3.9+
3. Se mantiene compatibilidad total con la implementación existente de boto3
4. Menor superficie de bugs: la lógica sync existente se preserva intacta

#### SessionManager: `threading.Lock` → `asyncio.Lock`

```python
# Antes
import threading
self._lock = threading.Lock()
def add_turn(self, ...):
    with self._lock:

# Después
import asyncio
self._lock = asyncio.Lock()
async def add_turn(self, ...):
    async with self._lock:
```

Los métodos de solo lectura (`get_history()`, `get_context_string()`) no requieren el lock porque las lecturas de dict en Python son seguras bajo asyncio (no hay parallelismo real).

## Justificación

### Alternativas consideradas

#### 1. Mantener sync con threadpool más grande

Aumentar el tamaño del threadpool de uvicorn (`--workers 4` o `--limit-concurrency`).

- **Pro**: Sin cambios en el código.
- **Contra**: Los threads son costosos en memoria (~5-10 MB c/u). Escalar a 100+ threads consumiría 500MB-1GB solo en overhead de threads. No resuelve el problema fundamental: threads bloqueados en I/O ociosa. GIL limita el paralelismo real.
- **Decisión**: Descartada. No escala y desperdicia recursos.

#### 2. Migrar a `aioboto3` para S3 async nativo

Usar `aioboto3` en lugar de `boto3` + `asyncio.to_thread`.

- **Pro**: Operaciones S3 truly async, mejor control de connection pooling.
- **Contra**: Nueva dependencia (~2MB), API ligeramente diferente a boto3, curva de aprendizaje. Para las operaciones actuales (descarga de un PDF), el overhead de `asyncio.to_thread` es despreciable.
- **Decisión**: Descartada por ahora. Se puede migrar en el futuro si se necesita connection pooling avanzado o streaming de archivos grandes.

#### 3. Usar `httpx` async en lugar de SDK de OpenAI

Reemplazar el SDK de OpenAI por llamadas directas con `httpx.AsyncClient`.

- **Pro**: Menos dependencia del SDK de OpenAI, más control sobre las llamadas.
- **Contra**: Se pierde el manejo automático de reintentos, parsing de respuestas, y tipado del SDK. Mayor código de boilerplate.
- **Decisión**: Descartada. El SDK de OpenAI ya proporciona `AsyncOpenAI` con la misma funcionalidad.

### Criterios de decisión

1. **Mínimo cambio de dependencias**: No agregar dependencias de runtime nuevas. `AsyncOpenAI` ya está incluido en `openai>=1.68.0`.
2. **Migración incremental**: Cada cliente se puede migrar independientemente sin romper el sistema.
3. **Compatibilidad de API**: `AsyncOpenAI` tiene la misma interfaz que `OpenAI`, minimizando cambios en la lógica de negocio.
4. **Fallback a sync**: Si una operación async falla, la lógica subyacente es la misma que la sync, facilitando debugging.

### Trade-offs aceptados

- **Complejidad del código**: `async/await` añade complejidad cognitiva comparado con código sync lineal. Mitigado por la simplicidad de la migración (mismo SDK, misma API).
- **Testing más complejo**: Tests async requieren `pytest-asyncio` y `AsyncMock`. Mitigado por la buena documentación de ambos.
- **S3 sigue parcialmente sync**: Las operaciones S3 usan `asyncio.to_thread` en lugar de truly async. Aceptable dado que S3 es rápido y no es el cuello de botella.

## Consecuencias

### Positivas

- **Event loop no se bloquea**: Durante los ~10s de espera de OpenAI, el event loop puede atender otros requests, health checks, y operaciones de la aplicación.
- **Menor uso de memoria**: Las coroutines de asyncio consumen ~0.1 MB vs ~5-10 MB por thread. Se pueden manejar cientos de requests concurrentes con la misma memoria que 40 threads.
- **Preparación para cola de mensajes**: Los clientes async son prerequisito para el sistema de workers descrito en ADR-008.
- **Sin nuevas dependencias**: `AsyncOpenAI` viene con `openai>=1.68.0` (ya en `pyproject.toml`). `asyncio.to_thread` es stdlib.

### Negativas

- **Código async "infecta" hacia arriba**: Una vez que un método es `async`, todos sus callers también deben ser `async`. Esto obliga a migrar el endpoint `/chat` y los tests.
- **Debugging más difícil**: Stack traces de asyncio son más difíciles de leer que stack traces síncronos.
- **Incompatibilidad con código sync existente**: No se puede llamar un método `async def` desde código síncrono sin `asyncio.run()` o `await`.

### Riesgos identificados

- **Riesgo**: `AsyncOpenAI` tiene comportamiento diferente a `OpenAI` en edge cases.
  **Mitigación**: La API del SDK está diseñada para ser idéntica. Los tests de integración con OpenAI real (ya existentes) verificarán la compatibilidad.

- **Riesgo**: `asyncio.to_thread` para S3 agrega overhead por creación de thread.
  **Mitigación**: Python reutiliza threads del default thread pool. El overhead es <1ms, despreciable comparado con la latencia de red de S3 (~200ms).

- **Riesgo**: Memory leak si coroutines no se completan (futures sin resolver).
  **Mitigación**: Implementar timeout en `await` de cada operación (ver ADR-008). Usar `try/finally` para cleanup.

## Notas adicionales

### Orden de migración

La migración debe seguir este orden para minimizar breakage:

1. **Clientes primero** (LLMClient, FileInputsClient, S3Client) — se pueden migrar manteniendo los tests aislados
2. **SessionManager** — necesario para que los clientes async puedan guardar sesiones
3. **Endpoint y workers** — integra todo, depende de los anteriores

### Compatibilidad con código existente

Durante la migración, puede existir un período donde algunos componentes son async y otros no. FastAPI permite mezclar `def` y `async def` endpoints. Los clientes sync pueden llamarse desde código async usando `asyncio.to_thread()`, aunque esto no se recomienda como estado final.

### Variables de entorno

No se requieren nuevas variables de entorno para esta decisión. La configuración de OpenAI (`OPENAI_API_KEY`) y AWS (`AWS_*`) se mantiene igual.
