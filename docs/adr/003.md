# ADR-003: File Inputs como método de retrieval

## Identificación

**Número**: ADR-003  
**Fecha**: 2026-03-20  
**Redactor**: @creep1ng, Ricardo Arias  
**Estado**: Aceptado

---

## Contexto

El chatbot requiere acceso al contenido de fichas técnicas de productos (paneles solares, inversores, controladores, baterías) con el fin de responder consultas técnicas precisas. Las fichas técnicas se encuentran almacenadas en formato PDF y contienen especificaciones técnicas detalladas. Un método de retrieval que garantice precisión en las respuestas y sea viable dentro del tiempo de desarrollo del proyecto es requerido.

El enfoque tradicional para este tipo de sistemas es implementar un pipeline RAG (Retrieval-Augmented Generation) con base de datos vectorial; sin embargo, dicha arquitectura presenta desafíos específicos para este caso de uso: alta similitud entre documentos (15 modelos de productos donde solo cambian un par de parámetros), costo de implementación significativo, y tiempo de desarrollo limitado.

---

## Decisión

Se adopta **File Inputs** (subida directa de PDFs al LLM) como método de retrieval para las fichas técnicas del chatbot.

El LLM (OpenAI, con soporte equivalente en Google, Anthropic y OpenRouter) procesa nativamente archivos PDF adjuntos, extrayendo texto y procesando imágenes integradas cuando es necesario. El chatbot invocará la herramienta `leer_ficha_tecnica(ruta)` que:

1. Descarga el PDF desde S3
2. Lo adjunta como File Input en la llamada al LLM
3. El LLM responde utilizando el contenido del documento como contexto

---

## Justificación

### Alternativas consideradas

1. **Vector DB + RAG (ChromaDB/Pinecone)**: Descartado porque:
   - La arquitectura RAG no funciona bien cuando existe alta similitud entre documentos, que es exactamente el caso del catálogo de productos de Arte Soluciones Energéticas (15 modelos donde solo un par de parámetros cambian entre ellos)
   - El tiempo de desarrollo requerido para implementar un pipeline RAG completo (chunking, embeddings, retrieval, evaluation) está fuera del alcance del MVP
   - El costo de tokens adicional por las queries de retrieval + generación de embeddings es superior al costo de adjuntar PDFs directamente en las llamadas al LLM

2. **Extracción local de texto + RAG**: Descartado porque:
   - No tiene sentido implementar extracción local si el provider de LLM lo hace de manera nativa y gratuita
   - Añadiría complejidad innecesaria sin beneficios tangibles

### Criterios de decisión

- **Precisión y trazabilidad**: File Inputs preserva 100% del contexto original del documento, evitando los errores de relevancia introducidos por RAG al tomar fragmentos
- **Simplicidad de implementación**: No requiere pipeline de chunking, modelo de embeddings, ni base de datos vectorial
- **Costo efectivo**: El incremento de tokens por adjuntar archivos es marginal comparado con el costo de implementar y mantener un sistema RAG completo, especialmente con el volumen actual de mensajes
- **Portabilidad**: Los principales providers (OpenAI, Google, Anthropic, OpenRouter) ofrecen funcionalidades equivalentes de File Inputs con diferentes opciones de OCR y reconocimiento de texto, por lo cual no representa lock-in técnico

### Trade-offs aceptados

- **Incremento de tokens por conversación**: Cada mensaje que requiere consultar una ficha técnica consume más tokens al adjuntar el PDF. El costo por conversación debe ser monitoreado.
- **Reevaluación futura**: Si el volumen de mensajes crece significativamente, puede ser necesario migrar a RAG para optimizar costos. Esta decisión debe ser revisada en Sprint 4.

## Consecuencias

### Positivas

- **Desarrollo acelerado**: El equipo puede centrarse en la lógica que genera valor (clasificación de intents, escalamiento, prompts de negocio) en lugar de malgastar tiempo en configuración de infraestructura de búsqueda vectorial
- **Precisión garantizada**: El LLM tiene acceso al documento completo, eliminando errores de retrieval que ocurrirían con RAG en documentos muy similares
- **Mantenimiento reducido**: No se requiere gestionar índices de embeddings, ni pipelines de re-indexación cuando las fichas técnicas son actualizadas
- **Portabilidad entre providers**: La funcionalidad de File Inputs está disponible en múltiples providers de LLM, permitiendo migrar si es necesario

### Negativas

- **Costo por token incrementally mayor**: Adjuntar PDFs a cada llamada incrementa el consumo de tokens. Para el volumen actual es rentable, pero debe monitorizarse
- **Límites de tamaño de archivo**: Los providers imponen límites en el tamaño de archivos adjuntos (típicamente 100-500 MB dependiendo del plan)
- **Decisión reversible**: Si el volumen de mensajes hace insostenible el costo de File Inputs, será necesario reimplementar RAG, lo cual representa deuda técnica

### Riesgos identificados

- **Riesgo**: Costo desproporcionado si el volumen de mensajes crece 10x o más
  **Mitigación**: Monitoreo de métricas de costo por sesión. Planificación de migración a RAG en Sprint 4 si el costo por conversación supera umbrales definidos.
- **Riesgo**: Límite de tamaño de archivo imposibilite adjuntar fichas técnicas muy grandes
  **Mitigación**: Normalización de PDFs a un tamaño máximo estándar durante el pipeline de ingesta. Límite documentado en README.

### Migración futura a RAG

Si en Sprint 4 se determina que File Inputs es demasiado costoso, la migración a RAG contemplará:

- Evaluación de embeddings especializados para documentos técnicos con alta similitud
- Estrategias de chunking que preserven las diferencias sutiles entre modelos
- Retrieval híbrido: File Inputs para productos específicos, RAG para consultas generales del catálogo

Esta decisión no está dentro del alcance actual pero está documentada para referencia futura.
