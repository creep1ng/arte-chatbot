# ADR-004: Estrategia de indexado y normalización del catálogo

## Identificación

**Número**: ADR-004  
**Fecha**: 2026-03-23  
**Redactor**: @creep1ng, Ricardo Arias  
**Estado**: Propuesto  

---

## Contexto

El pipeline de indexación del catálogo (TS-03, Issue #15) requiere una convención estandarizada para nombrar los archivos PDF de fichas técnicas y estructurar sus metadatos asociados. Esta convención debe satisfacer los siguientes requisitos:

- **Legibilidad humana**: Los nombres de archivo deben ser comprensibles para el equipo técnico y stakeholders.
- **Compatibilidad con S3**: Las rutas deben ser válidas en sistemas de archivos POSIX y en object storage (S3).
- **Unicidad garantizada**: No deben existir colisiones de nombre entre productos de diferentes categorías o fabricantes.
- **Integración con el System Prompt**: El índice (`catalog_index.json`) será inyectado en el System Prompt del LLM, por lo que los campos de metadatos deben ser concisos y autodescriptivos.
- **Facilitar Tool Calling**: El LLM debe poder identificar unívocamente un producto a partir de su nombre comercial para invocar `leer_ficha_tecnica(ruta)`.

Se definieron tres opciones iniciales durante la planificación del Sprint 2:

| Opción | Patrón | Ejemplo |
|--------|--------|---------|
| A | `{categoria}/{fabricante}-{modelo}.pdf` | `paneles/jinko-tiger-pro-460w.pdf` |
| B | `{categoria}_{fabricante}_{modelo}.pdf` | `panel_jinko_tiger-pro-460w.pdf` |
| **C** | **Nombre comercial exacto + JSON de metadatos** | `Jinko Tiger Pro 460W.pdf` + `catalog_index.json` |

---

## Decisión

Se adopta la **Opción C**: utilizar el **nombre comercial exacto del producto** como nombre del archivo PDF, acompañado de un archivo JSON de metadatos (`catalog_index.json`) que contiene información estructurada de cada producto.

### Convención de nombres de archivo

Los PDFs se almacenan en S3 con el siguiente patrón:

```
s3://{bucket}/raw/{categoria}/{nombre_comercial}.pdf
```

Donde:
- `{categoria}`: `paneles` | `inversores` | `controladores` | `baterias`
- `{nombre_comercial}`: Nombre de la serie o línea de productos, sin normalización agresiva, preservando mayúsculas, espacios y caracteres especiales del fabricante. Un mismo PDF puede contener fichas técnicas de múltiples variantes de la misma serie.

**Ejemplos concretos:**

```
s3://arte-chatbot-data/raw/paneles/Jinko Tiger Pro.pdf
s3://arte-chatbot-data/raw/inversores/Growatt SPA TL.pdf
s3://arte-chatbot-data/raw/controladores/Victron SmartSolar MPPT.pdf
s3://arte-chatbot-data/raw/baterias/Huawei LUNA2000.pdf
```

### Estructura del archivo de metadatos

El archivo `catalog_index.json` se almacena en:

```
s3://{bucket}/index/catalog_index.json
```

Su schema está definido en `docs/schemas/catalog_index.schema.json` y utiliza `oneOf` para validar que cada categoría tenga sus parámetros específicos. El schema implementa un patrón de **herencia con overrides**: los parámetros técnicos compartidos por todas las variantes de una serie se definen en `parametros_comunes` a nivel de producto, mientras que cada variante solo especifica en `parametros_clave` los campos que difieren (e.g., potencia, capacidad).

**Campos obligatorios (todas las categorías):**

| Campo | Tipo | Descripción | Ejemplo |
|-------|------|-------------|---------|
| `id` | `string` | Identificador único de la serie (slug) | `jinko-tiger-pro` |
| `nombre_comercial` | `string` | Nombre de la serie o línea de productos | `Jinko Tiger Pro` |
| `categoria` | `enum` | Categoría del producto | `paneles` |
| `fabricante` | `string` | Marca o fabricante | `Jinko` |
| `ruta_s3` | `string` | Ruta relativa en S3 | `raw/paneles/Jinko Tiger Pro.pdf` |
| `variantes` | `array` | Lista de variantes del producto, cada una con `modelo` y `parametros_clave` diferenciales | Cada item tiene `modelo` y `parametros_clave` |

**Campos opcionales (todas las categorías):**

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `parametros_comunes` | `object` | Parámetros técnicos compartidos por todas las variantes de la serie. Estructura definida por categoría |
| `descripcion` | `string` | Descripción breve del producto |
| `fecha_ingesta` | `string` | ISO 8601 timestamp de cuando se ingirió el documento |
| `version_ficha` | `string` | Versión de la ficha técnica si el fabricante la publica |
| `url_fabricante` | `string` | URL oficial del producto en el sitio del fabricante |

**Parámetros compartidos por categoría (`parametros_comunes`):**

| Categoría | Campos requeridos en `parametros_comunes` | Campos condicionales |
|-----------|-------------------------------------------|---------------------|
| `paneles` | `tipo_celda` | — |
| `inversores` | `tipo` | Si `onda_pura`/`onda_modificada`/`multifuncional`: `nivel_tension`. Si `hibrido`/`conexion_a_red`: `fases` |
| `controladores` | `tipo`, `tension_v` | — |
| `baterias` | `tipo` | Si `Litio`: `nivel_tension` |

**Parámetros diferenciales por variante (`parametros_clave`):**

| Categoría | Campo requerido en `parametros_clave` | Campos opcionales (override) |
|-----------|---------------------------------------|------------------------------|
| `paneles` | `potencia_w` | `tipo_celda` |
| `inversores` | `capacidad` | `tipo`, `nivel_tension`, `fases` |
| `controladores` | `capacidad_a` | `tipo`, `tension_v` |
| `baterias` | `capacidad` | `tipo`, `nivel_tension` |

## Justificación

### Alternativas consideradas

1. **Opción A (`{categoria}/{fabricante}-{modelo}.pdf`)**: Descartada porque:
   - Requiere normalización agresiva (lowercase, guiones) que pierde la presentación oficial del fabricante.
   - El modelo a veces incluye información redundante con el fabricante.
   - El LLM podría confundirse al intentar mapear "Jinko Tiger Pro 460W" (lenguaje natural) con `jinko-tiger-pro-460w` (slug normalizado).

2. **Opción B (`{categoria}_{fabricante}_{modelo}.pdf`)**: Descartada porque:
   - Similar a Opción A, requiere normalización que deshumaniza los nombres.
   - El prefijo de categoría es redundante dado que la carpeta padre ya indica la categoría.
   - Genera nombres muy largos con múltiples guiones bajos.

3. **Índice generado dinámicamente desde S3**: Descartada porque:
   - Acopla el pipeline de indexación al estado actual del bucket, introduciendo fragilidad.
   - No permite definir parámetros_clave sin parsear el PDF.
   - Requiere que cada consulta al chatbot liste el bucket completo (costo de API).

### Criterios de decisión

- **Fidelidad al nombre original**: Preservar el nombre comercial exacto facilita la correspondencia entre lo que dice el usuario ("¿Cuál es la potencia del Jinko Tiger Pro 460W?") y el archivo que el LLM necesita solicitar.
- **Separación de concerns**: El nombre del archivo es para identificación humana; los metadatos estructurados (categoría, fabricante, parámetros) viven en el JSON. Esto mantiene los PDFs inalterados y crea un índice consultable.
- **Compatibilidad con Tool Calling**: El LLM puede recibir el catálogo en el System Prompt y hacer matching semántico entre la consulta del usuario y `nombre_comercial` para determinar qué ficha técnica leer.
- **Facilidad de curación**: El equipo que cura las fichas técnicas (TASK-M2-01, Issue #44) puede copiar los PDFs con sus nombres originales sin necesidad de renombrarlos.

### Trade-offs aceptados

- **Espacios y caracteres especiales en nombres de archivo**: Algunos sistemas pueden tener problemas con espacios en rutas S3. Mitigación: S3 soporta espacios nativamente, y las URL se codifican automáticamente con `%20`.
- **Archivo JSON como punto único de falla**: Si `catalog_index.json` se corrompe, el catálogo queda inutilizable. Mitigación: versionado del archivo en S3 y backup en el repositorio.
- **Mantenimiento manual del JSON**: Cada nueva ficha técnica requiere una entrada manual en el JSON. Mitigación: script automatizado (TASK-M2-NEW1, Issue #75) que genera el índice a partir de los metadatos.

## Consecuencias

### Positivas

- **Correspondencia directa usuario-archivo**: El nombre comercial exacto elimina ambigüedad entre lo que el usuario consulta y lo que el LLM necesita.
- **PDFs inalterados**: No se renombran ni modifican los documentos originales, preservando su integridad.
- **Índice estructurado y consultable**: `catalog_index.json` permite búsquedas por categoría, fabricante o parámetros sin necesidad de parsear PDFs.
- **Facilita la generación automática**: El script de indexación (Issue #75) puede generar el JSON automáticamente a partir de los PDFs existentes en S3.
- **Extensibilidad**: Nuevos campos de metadatos pueden añadirse al JSON sin afectar la convención de nombres.

### Negativas

- **Gestión dual**: Se mantienen dos artefactos por producto: el PDF y su entrada en el JSON.
- **Riesgo de desincronización**: Si un PDF se elimina de S3 pero su entrada permanece en el JSON, el chatbot fallará al intentar leerlo.
- **Mayor complejidad en el pipeline de ingesta**: La generación del índice requiere un paso adicional (scraping de metadatos del PDF).

### Riesgos identificados

- **Riesgo**: Desincronización entre PDFs en S3 y entradas en `catalog_index.json`.
  **Mitigación**: Script de validación que verifica la existencia de cada `ruta_s3` referenciada en el JSON. Ejecutar como paso previo en CI.
- **Riesgo**: Colisión de nombres si dos fabricantes usan nombres similares.
  **Mitigación**: El campo `id` (slug) debe ser único y validarse durante la ingesta. Agregar prefijo de fabricante al slug si existe ambigüedad (`jinko-tiger-pro-460w` vs `trina-tiger-pro-460w`).
- **Riesgo**: Pérdida de metadatos si `catalog_index.json` se corrompe.
  **Mitigación**: Mantener una copia del JSON en `docs/data/catalog_index.json` dentro del repositorio como backup.

## Criterios de inclusión/exclusión de fichas técnicas

### Requisitos de legibilidad

| Criterio | Estado | Descripción |
|----------|--------|-------------|
| Texto extraíble | **Obligatorio** | El PDF debe contener texto seleccionable (no escaneos sin OCR). |
| Resolución mínima | **Obligatorio** | Mínimo 150 DPI para gráficos y tablas legibles. |
| Idioma | **Recomendado** | Español preferido. Inglés aceptable si no existe versión en español. |

### Requisitos de verificabilidad

| Criterio | Estado | Descripción |
|----------|--------|-------------|
| Especificaciones eléctricas | **Obligatorio** | Debe incluir al menos: potencia, voltaje, corriente. |
| Curvas o datos de rendimiento | **Recomendado** | Gráficos de eficiencia, curvas I-V (paneles), etc. |
| Datos del fabricante | **Obligatorio** | Logo, modelo y contacto del fabricante presentes. |

### Categorías válidas

Solo se ingieren fichas técnicas de las siguientes categorías:

1. **Paneles solares** (`paneles`): Módulos fotovoltaicos de cualquier tecnología (monocristalino, policristalino, thin-film).
2. **Inversores** (`inversores`): Inversores on-grid, off-grid e híbridos.
3. **Controladores de carga** (`controladores`): Reguladores MPPT y PWM.
4. **Baterías** (`baterias`): Baterías de litio, plomo-ácido y otras tecnologías de almacenamiento.

### Exclusiones explícitas

- Fichas de productos descontinuados (salvo que el stakeholder lo requiera).
- Documentos de marketing sin especificaciones técnicas.
- Manuales de instalación (a menos que contengan fichas técnicas completas).

## Notas adicionales

### Generación automática del índice

El pipeline de generación de `catalog_index.json` (Issue #75) debe:

1. Listar todos los PDFs en `s3://{bucket}/raw/`
2. Extraer metadatos básicos del PDF (título, autor, páginas)
3. Intentar extraer `parametros_clave` mediante parsing del contenido
4. Generar el campo `id` como slug del nombre comercial
5. Validar contra el schema JSON antes de publicar

### Versionado del schema

El schema JSON (`catalog_index.schema.json`) debe versionarse con el campo `$schema` apuntando a la URL del schema. Cambios al schema deben documentarse como breaking changes con migración.

### Integración con el System Prompt

El contenido de `catalog_index.json` (sin los PDFs) será inyectado en el System Prompt del LLM como contexto del catálogo disponible. La estructura debe optimizarse para legibilidad por parte del LLM, evitando campos anidados excesivos o metadatos innecesarios.

### Patrón de herencia con overrides

El schema implementa un patrón de herencia donde:

1. **`parametros_comunes`** (nivel producto): Define los parámetros técnicos que aplican a todas las variantes de la serie (e.g., tipo de celda, tipo de inversor, voltaje nominal).
2. **`parametros_clave`** (nivel variante): Solo especifica los parámetros que difieren entre variantes (e.g., potencia en watts, capacidad en amperios).
3. **Override opcional**: Una variante puede incluir en `parametros_clave` campos que ya existen en `parametros_comunes` para indicar que esa variante particular difiere del valor común.

Este patrón reduce la redundancia de datos cuando un mismo PDF de ficha técnica contiene múltiples variantes de la misma serie (e.g., un datasheet de "Victron SmartSolar MPPT" que cubre los modelos 150/35, 150/45 y 150/70, donde solo varía la capacidad en amperios).
