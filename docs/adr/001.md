# ADR-001: Arquitectura inicial y orquestación de servicios

**Número**: ADR-001

**Redactor**: Ricardo Arias  
**Estado**: Aceptado

## Contexto

El proyecto requiere una arquitectura de sistema reproducible, modular y escalable para el desarrollo del chatbot RAG. El equipo está compuesto por 4 estudiantes con conocimientos heterogéneos en infraestructura y distintas capacidades de hardware en sus máquinas locales. Los objetivos del Sprint 1 (Walking Skeleton) exigen:

- Un sistema funcional de punta a punta desde el primer sprint
- Reproducibilidad total del entorno de desarrollo entre integrantes
- Facilidad de integración continua (CI) con GitHub Actions
- Capacidad de aislar componentes para desarrollo independiente (backend, RAG, evaluación)
- Despliegue local sin depender de servicios cloud en esta fase académica

Según la planificación del proyecto[cite:1] y el issue TS-01[cite:3], el Sprint 1 debe establecer la infraestructura base con Docker y CI básico. Las features FEAT-01 (US-01: respuesta básica del LLM) y FEAT-02 (US-02: flag de escalamiento)[cite:5] requieren un backend funcional con integración a APIs de LLM.

---

## Decisión

Vamos a adoptar **Docker Compose** como herramienta de orquestación de servicios, **uv** como gestor de dependencias de Python, y definir la siguiente arquitectura de servicios para el Sprint 1:

- **Servicio `backend`**: API REST desarrollada con FastAPI que expone el endpoint `POST /chat` y `/health`. Integra la API de OpenAI/Anthropic para generación de respuestas básicas.
- **Servicio `chroma`**: Base de datos vectorial ChromaDB en modo servidor, lista para recibir embeddings desde el Sprint 2 cuando se active el pipeline RAG (TS-03, US-04).
- **Servicio `postgres`** (opcional en S1, obligatorio en S2): Base de datos relacional para persistir historial de conversaciones (`session_id`, mensajes, timestamps). En S1 puede implementarse como dict en memoria si PostgreSQL añade complejidad innecesaria.

La estructura de carpetas del repositorio será:

/backend        → código del servicio FastAPI
/rag            → módulos de ingesta, retrieval y chunking (inactivo en S1)
/evaluation     → harness de evaluación (TS-02)
/docs           → ADRs, diagramas, guías técnicas
/infra          → Dockerfile, docker-compose.yml, scripts de CI

Todos los servicios se levantan con un solo comando: `docker compose up`.

## Justificación

### Alternativas consideradas

1. **Poetry como gestor de dependencias**: Ampliamente usado, pero más lento que `uv` en resolución de dependencias. `uv` es compatible con pyproject.toml y significativamente más rápido (10–100x según benchmarks oficiales de Astral).
2. **pip-tools**: Ligero pero no maneja entornos virtuales nativamente. Requiere comandos adicionales (`pip-compile`, `pip-sync`) que añaden fricción.
3. **Despliegue nativo sin Docker**: Requeriría coordinar versiones de Python, variables de entorno y dependencias de sistema operativo entre 4 máquinas distintas. Incompatible con el objetivo de reproducibilidad.
4. **Kubernetes (minikube, kind)**: Sobrecarga de complejidad para un proyecto académico de 6 sprints. Docker Compose ofrece el 80% de los beneficios con el 20% de la complejidad.
5. **Qdrant en lugar de ChromaDB**: Qdrant es más robusto para producción, pero ChromaDB tiene mejor soporte de Python nativo y menor curva de aprendizaje. Para un prototipo académico con ≤ 50 fichas técnicas (US-09), ChromaDB es suficiente.

### Criterios de decisión

- **Simplicidad operacional**: El equipo necesita enfocarse en el pipeline RAG y las historias de usuario, no en troubleshooting de infraestructura. Docker Compose es el estándar de facto para orquestación local.
- **Velocidad de setup**: `uv` instala dependencias 10–100x más rápido que Poetry, crítico para CI en GitHub Actions y onboarding de nuevos colaboradores.
- **Alineación con Walking Skeleton**: ChromaDB en S1 está inactivo pero presente, reduciendo el delta de cambios en S2 cuando se active el RAG (TS-03).
- **Conocimiento del equipo**: Ricardo tiene experiencia en Docker y administración de servidores. Esto minimiza el riesgo de bloqueos técnicos.

### Trade-offs aceptados

- **PostgreSQL como "overkill" en S1**: Si la gestión de sesiones con dict en memoria es suficiente para el alcance del sprint, PostgreSQL puede retrasarse a S2. Esto reduce la complejidad inicial pero obliga a migrar datos más adelante.
- **ChromaDB no es producción-ready**: Para el alcance académico es adecuado, pero si el proyecto escala a producción, migrarlo a Qdrant o Pinecone requerirá reescribir el módulo de retrieval.
- **Dependencia de APIs externas (OpenAI/Anthropic)**: El sistema no es 100% self-hosted porque el LLM se consume vía API. Alternativas como Ollama (LLM local) son posibles pero añaden latencia y complejidad de configuración que no justifican en esta fase.

## Consecuencias

### Positivas

- **Reproducibilidad garantizada**: Cualquier integrante puede correr `docker compose up` y tener el sistema funcional en < 5 minutos.
- **CI simple**: GitHub Actions puede ejecutar el mismo comando sin configuración adicional de runners.
- **Modularidad**: Cada servicio puede desarrollarse, testearse y desplegarse de forma independiente. El backend puede evolucionar sin tocar ChromaDB y viceversa.
- **Preparación para S2**: La arquitectura ya contempla el servicio ChromaDB, reduciendo el trabajo de TS-03 (pipeline de ingesta) a solo poblar la base de datos vectorial.
- **Velocidad de desarrollo**: `uv` reduce el tiempo de instalación de dependencias en CI de ~2 minutos (Poetry) a ~10 segundos, permitiendo iteraciones más rápidas.

### Negativas

- **Overhead de Docker en máquinas con recursos limitados**: Si algún integrante tiene hardware antiguo, Docker Desktop puede consumir 2–4 GB de RAM. Mitigación: permitir desarrollo nativo con `uv` para componentes específicos (ej. módulo de evaluación).
- **Curva de aprendizaje inicial**: Integrantes sin experiencia en Docker necesitarán tiempo para entender volúmenes, redes y comandos básicos. Mitigación: documentar en `README.md` los comandos críticos y casos de uso comunes.
- **Migración futura si se escala a Kubernetes**: Si el proyecto necesita desplegarse en producción con alta disponibilidad, los `docker-compose.yml` no son directamente traducibles a manifiestos de Kubernetes. Mitigación: esto está fuera del alcance académico (ver Entregable 5: recomendaciones de producción).

### Riesgos identificados

- **Riesgo**: Problemas de conectividad de red entre servicios en Docker (ej. backend no puede alcanzar ChromaDB).
  **Mitigación**: Usar nombres de servicio como hostnames en la configuración (`chroma:8000`), que Docker Compose resuelve automáticamente.
- **Riesgo**: Conflictos de versiones de Python entre servicios si se usan imágenes base distintas.
  **Mitigación**: Estandarizar en `python:3.13-slim` para todos los servicios.
- **Riesgo**: Pérdida de datos de ChromaDB si el contenedor se recrea sin volúmenes persistidos.
  **Mitigación**: Configurar volúmenes nombrados en `docker-compose.yml` para `/chroma/data`.

## Decisión de Testing: Pytest como Framework

### Nombre de la decisión

**Pytest como framework de testing**

### Justificación

Se ha elegido **pytest** como framework de testing para el proyecto por las siguientes razones:

1. **Simplicidad**: Pytest requiere mínima configuración y permite escribir tests con sintaxis Python natural, utilizando funciones que empiezan con `test_`.
2. **Fixtures**: El sistema de fixtures de pytest permite gestionar dependencias y estados de manera limpia y reutilizable.
3. **Extensibilidad**: Amplio ecosistema de plugins (pytest-cov, pytest-asyncio, pytest-mock) que se integran sin configuración adicional.
4. **Integración con CI**: Compatible nativamente con GitHub Actions y otros sistemas de integración continua.
5. **Comunidad**: Amplia adopción en proyectos Python modernos, incluyendo frameworks como FastAPI, Django y SQLAlchemy.

### Estructura de directorios sugerida

Los tests se organizarán en carpetas `tests/` dentro de cada módulo principal, siguiendo el patrón:

```
backend/tests/
├── __init__.py
├── test_api.py
└── test_models.py

rag/tests/
├── __init__.py
├── test_retriever.py
└── test_generator.py
```

Cada módulo contendrá:
- `__init__.py`: Archivo vacío para marcar el directorio como paquete Python.
- `test_*.py`: Archivos de test que agrupan tests relacionados por funcionalidad.

### Comandos de ejecución

```bash
# Ejecutar todos los tests
pytest

# Ejecutar tests con output detallado (-v)
pytest -v

# Ejecutar tests con coverage (requiere pytest-cov)
pytest --cov=backend --cov=rag

# Ejecutar tests específicos de un módulo
pytest backend/tests/

# Ejecutar tests con marcadores específicos
pytest -m "not integration"
```

### Ejecución en entorno de desarrollo

Para ejecutar los tests en el entorno local:

```bash
# Desde la raíz del proyecto (sin Docker)
pytest

# Dentro del contenedor del backend
docker compose exec backend pytest

# Con coverage en el contenedor
docker compose exec backend pytest --cov=backend --cov-report=term-missing
```

### Dependencias requeridas

Las dependencias de testing se especifican en los archivos `requirements.txt` de cada módulo:

```txt
pytest>=8.0.0
pytest-cov>=4.0.0
pytest-asyncio>=0.23.0
```

### Trade-offs aceptados

- **pytest-asyncio para tests asíncronos**: Los tests de endpoints FastAPI son asíncronos. Se requiere `pytest-asyncio` para ejecutar correctamente los tests del backend.
- **Coverage opcional en CI**: El coverage puede no alcanzar el 100% inicialmente. Se aceptará coverage parcial siempre que los tests cubran las funcionalidades críticas.

## Notas adicionales

### Decisión sobre PostgreSQL en S1

Si durante la implementación de US-01 se determina que gestionar `session_id` con un dict en memoria (`sessions = {}`) es suficiente para cumplir los criterios de aceptación, PostgreSQL puede omitirse en el `docker-compose.yml` inicial y añadirse en S2 cuando US-06 (contexto básico de sesión) requiera persistencia real.

### Consideración de costos de API

El presupuesto estimado para llamadas a OpenAI/Anthropic en fase de desarrollo es ~USD 50–100 para los 6 sprints (basado en 500–1000 queries de prueba $\times$ 0.01–0.10 USD/query según modelo). Si el costo se vuelve prohibitivo, la alternativa es migrar a Ollama con Llama 3 o Mistral localmente, aceptando aumento de latencia de 2–3s a 10–15s.
