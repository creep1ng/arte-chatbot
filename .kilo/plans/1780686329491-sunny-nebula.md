# Plan: Replace Per-Service Fargate Tunnels With a Low-Cost EC2 Compose Host

## Recommendation

Migrate the current production exposure model from three ECS/Fargate services, each with its own `cloudflared` sidecar, to one small EC2 instance running Docker Compose and one Cloudflare Tunnel that routes all public hostnames to local containers.

This keeps internet exposure cheap without ALB or NAT Gateway:

- One EC2 instance in a public subnet with outbound internet access.
- No public inbound application ports.
- No NAT Gateway.
- One `cloudflared` connector/container for `chatbot`, `app`, and `admin` hostnames.
- Existing ECR repositories remain the image source.
- Existing Cloudflare DNS/proxy remains the public entrypoint.

## Why This Is Cheaper

The current Terraform creates three Fargate services in `infra/terraform/envs/prod/main.tf`, each with:

- An app container.
- A colocated `cloudflared` sidecar.
- Separate Cloudflare Tunnel resources and tunnel-token secrets.
- Private subnet networking that requires either NAT or enough VPC endpoints for outbound AWS/Cloudflare dependencies.

For the current scale, this pays the always-on compute cost three times. A single EC2 instance consolidates those services into one host and one tunnel process. It trades horizontal scalability and managed scheduling for much lower fixed cost, which matches the stated goal.

## Target Architecture

```text
Cloudflare DNS / Proxy
        |
        v
Single Cloudflare Tunnel
        |
        v
EC2 instance, public subnet, no inbound app ports
        |
        +-- cloudflared container
        +-- backend container  -> port 8000
        +-- frontend container -> port 3000
        +-- admin container    -> port 3000
```

Cloudflare Tunnel ingress should route multiple hostnames through one tunnel:

- `chatbot.<domain>` -> `http://backend:8000`
- `app.<domain>` -> `http://frontend:3000`
- `admin.<domain>` -> `http://admin:3000`
- fallback -> `http_status:404`

Use Docker Compose service DNS names, not `localhost`, because `cloudflared` will be a separate container on the same Compose network.

## Terraform Changes

### 1. Add a Single-Tunnel Cloudflare Configuration

Update or reuse `infra/terraform/modules/cloudflare_tunnel` so production creates one tunnel instead of three.

Current production modules to replace:

- `backend_cloudflare_tunnel`
- `frontend_cloudflare_tunnel`
- `admin_cloudflare_tunnel`
- `cloudflare_tunnel_secrets` containing three tunnel tokens

New production module:

- `app_cloudflare_tunnel` or `edge_tunnel`
- `public_hostnames` includes backend, frontend, and admin routes
- one tunnel token secret, for example `cloudflare_tunnel_token`

### 2. Add an EC2 Compose Module

Create a Terraform module such as `infra/terraform/modules/ec2_compose_host` with:

- `aws_instance`
- `aws_security_group`
- `aws_iam_role`, `aws_iam_instance_profile`, and inline policies
- optional `aws_cloudwatch_log_group` only if needed later
- `user_data` or cloud-init bootstrap script

Instance placement:

- Use a public subnet variable, not `private_subnet_ids`.
- Associate a public IP for outbound internet access.
- Do not open ports `8000`, `3000`, `80`, or `443` to the world.
- Prefer no inbound SSH; use AWS Systems Manager Session Manager.
- If SSH is required temporarily, restrict it to a configured admin CIDR.

IAM instance role needs:

- ECR pull permissions for the three repositories.
- S3 read/list permissions for the backend catalog bucket.
- Secrets Manager/SSM read permissions for runtime secrets and Cloudflare tunnel token.
- SSM managed-instance permissions if Session Manager is used.

### 3. Bootstrap the Host

Use cloud-init/user data to install and configure:

- Docker Engine or distro Docker package.
- Docker Compose plugin.
- AWS CLI if not already available.
- A deployment directory, for example `/opt/arte-chatbot`.
- A generated `docker-compose.yml` using ECR image URIs and immutable tags.
- A generated deploy script, for example `/opt/arte-chatbot/deploy.sh`.
- A systemd unit that runs the Compose stack on boot.

Compose stack should include:

- `backend`
- `frontend`
- `admin`
- `cloudflared`

The `cloudflared` service should use the single tunnel token fetched from Secrets Manager or SSM.

Runtime secrets:

- Fetch backend secrets at deploy/start time from Secrets Manager or SSM.
- Write them to a root-owned env file with `0600` permissions, or inject them through the systemd unit.
- Avoid rendering secret values directly into Terraform state.

### 4. Replace Production ECS Resources

In `infra/terraform/envs/prod/main.tf`:

- Keep ECR modules.
- Keep URL locals and CORS locals.
- Remove `aws_ecs_cluster`.
- Remove `backend_service`, `frontend_service`, and `admin_service` ECS modules.
- Remove per-service Cloudflare tunnel modules.
- Add the single Cloudflare tunnel module.
- Add the EC2 Compose host module.

In `infra/terraform/envs/prod/variables.tf`:

- Replace `private_subnet_ids` with a `public_subnet_id` or `public_subnet_ids` input.
- Remove Fargate-only variables: `task_cpu`, `task_memory`, `desired_count`, `assign_public_ip`, and per-service tunnel secrets.
- Add EC2 variables: `instance_type`, `ami_id` or `ami_filter`, optional `ssh_allowed_cidr`, and maybe `root_volume_size_gb`.
- Add one tunnel secret input if the module still requires a static tunnel secret.

In `infra/terraform/envs/prod/outputs.tf`:

- Replace ECS outputs with EC2 outputs: instance id, public IP, public URLs, ECR repository URLs, and optional deploy role ARN.

### 5. Update GitHub Deploy Flow

Current `.github/workflows/ci.yml` deploys production by applying Terraform with new image tags, which updates ECS task definitions and services.

Replace the production deploy behavior with one of these minimal options:

Preferred low-cost flow:

1. Continue building and pushing immutable ECR images.
2. Terraform owns infrastructure, not every deploy.
3. GitHub Actions uses AWS SSM Run Command against the EC2 instance.
4. The command runs `/opt/arte-chatbot/deploy.sh sha-${{ github.sha }}`.
5. The script logs into ECR, updates image tags/env, pulls images, and runs `docker compose up -d`.

This avoids Terraform apply on every app deploy and keeps infra/app deployment responsibilities cleaner.

Deploy role changes:

- Remove ECS deploy permissions.
- Add `ssm:SendCommand`, `ssm:GetCommandInvocation`, and read access to the target instance metadata as needed.
- Keep ECR push permissions.

### 6. Decide What To Do With Local Staging

Production should be migrated first.

For `infra/terraform/envs/local-staging`, choose one of two follow-ups:

- Leave it temporarily on ECS/Fargate if local staging is ephemeral and already working.
- Later apply the same EC2 Compose pattern if staging cost is also a problem.

Do not migrate both production and staging in the same first pass unless cost pressure requires it. That would increase risk without improving the core production decision.

## Implementation Order

1. Create or adapt a single Cloudflare Tunnel module usage for multiple hostnames.
2. Add the `ec2_compose_host` Terraform module.
3. Refactor `envs/prod` to keep ECR and replace ECS/Fargate with EC2 Compose.
4. Add host bootstrap/deploy templates managed by Terraform.
5. Update GitHub OIDC/deploy IAM permissions from ECS update to SSM Run Command.
6. Update `.github/workflows/ci.yml` production deploy job to call the EC2 deploy script through SSM.
7. Run Terraform validation and formatting.
8. Apply in production during a planned maintenance window.
9. Verify public URLs and backend health through Cloudflare.
10. Remove old ECS/Fargate resources only after confirming the EC2 stack is serving traffic.

## Validation

Run locally:

```bash
terraform -chdir=infra/terraform/envs/prod fmt -recursive
terraform -chdir=infra/terraform/envs/prod validate
```

After apply/deploy:

```bash
curl -f https://chatbot.<domain>/health
curl -f https://app.<domain>/
curl -f https://admin.<domain>/
```

Also verify:

- Cloudflare DNS records point to the single tunnel.
- Only one tunnel token is required.
- EC2 security group does not expose app ports publicly.
- Instance can pull from ECR.
- Backend can read S3 catalog data.
- Runtime secrets are not stored in Terraform state as plaintext.

## Tradeoffs

Accepted tradeoffs:

- Less managed than Fargate.
- One host is a single point of failure.
- Deployments depend on instance health.
- Secrets may exist on disk unless injected more strictly.

Why acceptable now:

- The user explicitly does not need scaling yet.
- The priority is keeping fixed monthly cost low.
- Cloudflare Tunnel preserves public HTTPS exposure without ALB.
- Session Manager can preserve admin access without opening SSH.

## Out Of Scope For First Pass

- ALB, NLB, NAT Gateway, or ECS Service Connect.
- Multi-AZ high availability.
- Autoscaling.
- Blue/green deployment.
- Full secrets sidecar/agent implementation.
- Migrating local-staging unless requested separately.
