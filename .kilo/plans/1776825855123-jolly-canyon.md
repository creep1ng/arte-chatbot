## 1. ¿Qué hace el Evaluation Harness actualmente?

El evaluation harness es un conjunto de scripts que miden diferentes aspectos del chatbot:

### 1.1 Main Harness (`evaluation/harness/run.py`)
- **Qué hace:** Envía 5 queries al endpoint `/chat` y mide latencia, respuesta, session_id, escalate flag, y sources consultadas
- **Output:** JSON y CSV en `evaluation/harness/output/`
- **Dataset:** `evaluation/harness/dataset.json` (5 queries)
- **Métricas:** latency (min/avg/max), successful vs failed

### 1.2 Intent Evaluation (`evaluation/intent_eval/run_eval.py`)
- **Qué hace:** Envía 15 queries clasificadas por tipo (FAQ, product_info, escalate_quote, escalate_technical, escalate_order) y mide accuracy vs threshold 80%
- **Output:** JSON en `evaluation/intent_eval/output/intent_eval_<timestamp>.json`
- **Dataset:** `evaluation/intent_eval/dataset.json` (15 queries)
- **Métricas:** Accuracy %, per-category breakdown
- **Exit code:** 0 si pasa (>=80%), 1 si falla

### 1.3 Escalation Evaluation (`evaluation/escalation_eval/run_eval.py`)
- **Qué hace:** Envía 20 queries con flags `should_escalate` y mide False Positive Rate (threshold 15%) y False Negative Rate (threshold 10%)
- **Output:** JSON en `evaluation/escalation_eval/output/escalation_eval_<timestamp>.json`
- **Dataset:** `evaluation/escalation_eval/dataset.json` (20 queries)
- **Métricas:** TP, TN, FP, FN rates
- **Exit code:** 0 si ambos thresholds cumplen, 1 si falla

### 1.4 Hallucination Check (`evaluation/hallucination_check.py`)
- **Qué hace:** Analiza el CSV del main harness buscando:
  - `num_sources == 0` + valores técnicos en respuesta (W, V, A, %, °C, kWh)
  - Valores numéricos en respuesta que no aparecen en sources consultadas
- **Output:** JSON en `evaluation/results/hallucination_<timestamp>.json`
- **Dataset:** CSV del main harness
- **Métricas:** Hallucination rate % (threshold: <20%)
- **Opcional:** Upload a S3 (`--upload-s3` flag)

### 1.5 Human Evaluation (`evaluation/human_eval/cli.py`)
- **Qué hace:** CLI interactiva para evaluación manual con rúbrica de 6 dimensiones
- **Output:** `evaluation_results.json`
- **Métricas:** Weighted score >= 3.5/5, escalation accuracy >= 90%

## 2. Problema: ¿Qué se hace con los resultados?

**Situación actual:**
- Cada script genera JSON/CSV localmente
- Solo `hallucination_check.py` tiene opción de subir a S3 (`--upload-s3`)
- **No hay automatización** que:
  - Compare con runs anteriores (baseline)
  - Postee resultados en PRs
  - Bloquee merge por umbrales
  - Envíe notificaciones
  - Actualice dashboard

**Resultado neto:** El evaluation harness es un gasto de tokens (LLM calls) que genera archivos que probablemente nadie revisa después del CI run.

## 3. Manejo de Resultados del Evaluation Harness

**Decisión:** Archival simple en S3 en la ruta requerida por el notebook `quality_report.ipynb`.

**Estructura de S3 requerida:**
```
arte-chatbot-data/
└── evaluation/
    └── reports/
        └── {sprint}/
            ├── harness/report_YYYYMMDD_HHMMSS.json
            ├── hallucination/report_YYYYMMDD_HHMMSS.json
            └── human_eval/report_YYYYMMDD_HHMMSS.json
```

**Flujo en CI:**
1. Ejecutar el evaluation harness con el flag `--upload-s3`.
2. Asegurar que las variables de entorno de AWS (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_BUCKET_NAME`) estén configuradas en el workflow del CI para permitir la subida.

## 4. CI/CD: Publicación de Imágenes a Registry

**Decisión:** **Amazon ECR** (Elastic Container Registry).

**Justificación:** Aunque `ghcr.io` puede usarse con AWS Fargate y Lambda, si el repositorio (o la imagen) es privado, requiere configurar autenticación en AWS (ej. almacenar el token de GitHub en AWS Secrets Manager y referenciarlo en el Task Definition de Fargate). Al usar **AWS ECR**, la integración con IAM es nativa y el despliegue a Fargate o Lambda no requiere gestionar credenciales adicionales en los servicios de cómputo, simplificando drásticamente el setup de infraestructura.

**Flujo propuesto:**
```yaml
- name: Configure AWS credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
    aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    aws-region: us-east-1

- name: Login to Amazon ECR
  id: login-ecr
  uses: aws-actions/amazon-ecr-login@v2

- name: Build and push to ECR
  uses: docker/build-push-action@v6
  with:
    push: ${{ github.ref == 'refs/heads/main' }}
    tags: ${{ steps.login-ecr.outputs.registry }}/arte-chatbot-backend:${{ github.sha }}
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

## 5. CI/CD: Builds Diferenciales e Incrementalidad

**Problema actual:** `--no-cache` reconstruye todo desde cero.

**Soluciones:**
1. **Docker layer caching:** Usar `cache-from: type=gha` y `cache-to: type=gha,mode=max`.
2. **Builds Diferenciales:** Detectar qué servicios cambiaron usando Git.
   ```bash
   git diff --name-only HEAD~1 | grep -q "^backend/" && rebuild_backend=true
   ```

## 6. Resolución de Error en CI (Missing API Key)

**Problema reportado:**
```
Query q001 failed: HTTP 401: {"detail":"Missing API key"}
```
Este error ocurre porque el contenedor del backend en el CI o el job de evaluation harness no están recibiendo la variable de entorno `CHAT_API_KEY` (ni `OPENAI_API_KEY`), lo cual provoca un HTTP 401 al hacer requests al endpoint `/chat`.

**Solución:**
En `.github/workflows/ci.yml`, pasar los secrets al step de docker-compose y al job de evaluation:
```yaml
env:
  CHAT_API_KEY: ${{ secrets.CHAT_API_KEY }}
  OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```
