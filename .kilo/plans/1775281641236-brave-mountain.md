### Fase 1: Módulo de Cola de Mensajes (Issue #101)

**Archivo**: `backend/app/queue.py`

1. **Dataclass `ChatMessage`**
   - `request_id: str`
   - `session_id: str`
   - `message: str`
   - `future: asyncio.Future` (para resolver resultado o excepción)

2. **Clase `MessageQueue`**
   - Constructor con parámetros:
     - `max_workers: int = 5`
     - `max_queue_size: int = 100`
     - `llm_client, s3_client, file_inputs_client, session_manager` (inyección de dependencias)
   
   - Métodos:
     - `async start()`: Crea y lanza `max_workers` tareas asyncio
     - `async stop()`: Cancela todos los workers limpiamente
     - `async enqueue(message: ChatMessage)`: Encola mensaje con backpressure
     - `async _worker(worker_id: int)`: Loop principal del worker
       1. `await self.queue.get()`
       2. Procesar mensaje (escalation check, LLM calls, tool calls)
       3. `message.future.set_result()` o `.set_exception()`
       4. `self.queue.task_done()`

3. **Helpers privados en MessageQueue**
   - `async _process_tool_call_async()`: Lógica de tool calling (move from main.py)

4. **Tests** (`backend/tests/test_queue.py`)
   - Tests de creación, start/stop lifecycle
   - Tests de enqueue/dequeue
   - Tests de procesamiento de mensajes con mocks
   - Tests de manejo de errores
   - Tests de backpressure
   - Tests de concurrencia (múltiples workers)

## Cambios de Arquitectura Clave

### Flujo Actual (Con Cola)
```
Cliente HTTP
    ↓
/chat endpoint (async)
    ↓
Encolar ChatMessage con asyncio.Future
    ↓
Esperar max 60s en future
    ↓
Worker async procesa en paralelo
    ├→ Escalation check
    ├→ LLM call (async)
    ├→ Tool calls (async)
    ├→ Guardar en SessionManager (async with lock)
    └→ Resolver future con resultado
    ↓
Cliente recibe respuesta
```

### Cambios de Interfaces

1. **SessionManager**
   - `add_turn()` ya no es sync, requiere `await`
   - `clear_session()` ya no es sync, requiere `await`

2. **LLMClient, FileInputsClient, S3Client**
   - Métodos públicos ya son async (PR #108 ya los tiene)

3. **MessageQueue (nuevo)**
   - Inyecta todos los clientes y managers
   - Gestiona pool de workers
   - Encola y resuelve futures

## Decisiones Técnicas

### Por qué `asyncio.to_thread()` en S3Client (ya implementado en PR #108)
- S3 es rápido (~100-500ms), no justifica dependencia adicional `aioboto3`
- `asyncio.to_thread()` usa threadpool automático de FastAPI/asyncio

### Por qué no cambiar dependencias existentes
- Mantener compatibilidad hacia atrás
- Solo agregar async wrappers donde sea necesario

### Timeout de 60 segundos en futures
- Corresponde al typical timeout de OpenAI API calls (~20-40s)
- Con 60s tenemos buffer para errores de red
